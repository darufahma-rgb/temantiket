Fix duplicate and orphan agent commission/wallet transactions in Temantiket.

Context:
From shell grep, the suspicious files are:
- src/components/VisaEntryPanel.tsx
- src/lib/agentWallet.ts
- src/lib/agentFeeBreakdown.ts
- src/features/orders/ordersRepo.ts
- src/features/clients/clientsRepo.ts

Current bug:
1. Agent page shows 2 commission entries for the same order.
2. Same order ID appears twice with the same Rp200.000 fee.
3. Deleted client/order still leaves Rp400.000 commission visible.
4. This means wallet/commission rows are duplicated and orphaned.

Likely root cause:
- Commission insert is not idempotent.
- agent_wallet_transactions does not prevent duplicates.
- deleteOrder/deleteClient does not clean up related wallet transactions.
- UI totals count orphaned wallet transactions.

Goal:
Make agent wallet/commission ledger accurate and safe.

Tasks:

1. Audit commission creation flow:
   - Search for creation of wallet transactions with types:
     - voa_agent_fee
     - field_agent_fee
     - pelaksana_fee
     - mission_fee
     - order_bonus
     - kurir_fee
     - operational_fee
   - Find where VisaEntryPanel credits VOA / pelaksana fee.
   - Find any status-change effect that inserts wallet records.

2. Make wallet transaction creation idempotent:
   - Add a helper in src/lib/agentWallet.ts:
     createWalletTxOnce(...)
   - Before inserting, check whether a transaction already exists for:
     agencyId
     agentId
     orderId
     type
   - If exists, return existing transaction instead of inserting a duplicate.

3. Add database-level protection:
   - Add Supabase migration for unique protection.
   - Recommended unique key:
     agency_id, agent_id, order_id, type
   - Only apply when order_id is not null.
   - Example:
     CREATE UNIQUE INDEX IF NOT EXISTS uniq_wallet_tx_order_fee
     ON agent_wallet_transactions (agency_id, agent_id, order_id, type)
     WHERE order_id IS NOT NULL;

4. Fix existing duplicate data:
   - Add safe SQL repair migration/script:
     - find duplicates by agency_id, agent_id, order_id, type
     - keep the oldest or newest valid row
     - delete the rest
   - Do not delete payout rows.
   - Do not delete mission rows without order_id.

5. Fix orphan transactions:
   - Any wallet transaction with order_id pointing to a missing order should not count in active totals.
   - Either:
     A. delete orphan wallet rows safely
     OR
     B. mark them reversed/voided
     OR
     C. filter them out from UI totals
   - Prefer safe cleanup + UI filtering.

6. Fix delete flow:
   - When deleteOrder(id) succeeds, also remove or void wallet transactions linked to that order_id.
   - When deleteClient(id) deletes related orders, ensure related wallet transactions are also cleaned.
   - Do not leave orphan commission rows.

7. Fix UI total calculation:
   - In agent fee breakdown / wallet balance:
     - count only valid active wallet rows
     - do not count duplicate order fee rows
     - do not count rows whose order was deleted
   - Make "Komisi Lapangan" assignment count use unique order/task IDs, not raw duplicated rows.

8. Add visible debug protection:
   - If duplicate rows are detected in UI data, log a warning in console.
   - Do not show duplicated rows in agent profile.

9. Validate:
   - Create 1 VOA order assigned to 1 agent.
   - Mark order as selesai.
   - Confirm only 1 wallet tx is created.
   - Refresh page.
   - Confirm it does not create another tx.
   - Delete order/client.
   - Confirm commission disappears or is voided.
   - Confirm total is no longer Rp400.000 for one order.
